# ADR-0024: gearbox-update --refresh-drift — Recopy Drifted Copies + Full Reference Mapping

- Date: 2026-07-20
- Status: accepted
- Related: ADR-0017 (gearbox-update), ADR-0021 (the hash stamp contract), ADR-0023 (the version number)

## Context

Issue #51 (maintainer session instruction). Two gaps:

1. **Drift is only reported, never fixed**: after upstream revises an already-copied ADR, `gearbox-version` reports ⚠️ drift (ADR-0021), but for a slug-matched existing file, `gearbox-update` always skips it — the loop breaks at "manual diff or recopy."
2. **A reference-mapping bug in incremental backfills**: `transformAdrContent`'s renumberMap only contains the number mappings for **this run's newly copied** files, not the pairings from historical renumbering. A first full backfill (copying everything in one batch) never hits this, but an incremental backfill does: a newly copied ADR's body references to an older entry keep the upstream number, pointing at the wrong file downstream. Confirmed case: on Blackbox's `docs/gearbox-backfill-2026-07-19` branch, the newly copied 0023 (install)'s body referenced `ADR-0016`, which in Blackbox is actually 0017.

The maintainer once asked "can this be a mirror sync (delete what downstream has extra, add what's missing)" — **rejected**: downstream's `docs/adr/` is a mix of "upstream copies + downstream's own decisions"; deleting the "extras" would destroy downstream's own ADRs; and backfill is a prompt, not a mandate (ADR-0013) — downstream has the right to decline, and mirroring would force back in what was already declined. Sync semantics stay one-directional and incremental; this ADR only closes the drift loop.

## Decision

1. **`--refresh-drift` flag (opt-in)**: for a slug-matched copy carrying a `sha256:` stamp, compare the recorded hash against upstream's current hash; on drift → rerun the transformation on upstream's latest content and overwrite (keeping downstream's own numbering; the provenance line is regenerated by the transformation, and the hash stamp is refreshed automatically), each recopy is its own commit, on the same backfill branch.
   - **Not touched**: downstream's own ADRs (no upstream slug/provenance pairing), legacy copies with no stamp (drift is not detectable, per ADR-0021's boundary), provenance-by-reference sync (not a copy).
   - **Known trade-off (accepted)**: overwriting discards any local edits downstream made to the copied file. The backstop is branch + PR review (the diff is visible) + downstream's L1 process; hence it's a flag, not default behavior.
   - Without the flag, detecting drift → the skip reason explicitly reports the drift and points at the flag; no longer silent.
2. **Reference-mapping fix (unconditional)**: the mapping table used for transformation = **all existing slug/provenance pairings** (upstream number → downstream number) ∪ this run's newly assigned numbers. Shared by both new copies and recopies.

## Consequences

- Drift loop closed: version reports ⚠️ → update --refresh-drift recopies → the stamp updates → version turns ✅
- Cross-references in incremental backfills are now correct going forward; Blackbox's existing backfill branch is recommended for abandonment and rerunning (the fixed tool will produce correct references)
- The risk of overwriting local edits is made explicit (flag + review), without introducing "downstream-edit protection" — a complex diff-merge mechanism; this is YAGNI, revisit if a real need shows up
- Tiering: purely tool behavior (the ADR-0017 precedent of "a personal tool, not a protocol change"), does not touch AGENTS.md; the standard three-piece flow still applies (issue #51 + this ADR + a PR), CI green enables self-merge; Version bump: minor (a new mechanism)
